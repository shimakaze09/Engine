# AGENTS.md

## Project Identity

This repository is a C++20 game engine.

The project is not production-complete yet. Treat it as an actively evolving engine with editor, runtime, renderer, physics, audio, scripting, asset, tooling, and test systems.

Primary goals:

* Build a modular game engine from scratch.
* Keep engine contributors working mainly in C++.
* Keep game authors working mainly through Lua scripts and editor workflows.
* Preserve predictable runtime behavior.
* Keep dependencies explicit, minimal, and directed downward.
* Keep the project friendly to clangd, Codex, and other code-intelligence tools.

## Required Reading Before Work

Before making non-trivial changes, read these files first:

1. `README.md`
2. `PROJECT_INDEX.md`, if it exists
3. `AGENTS.md`
4. `.clangd`, if it exists
5. Root `CMakeLists.txt`
6. The relevant module `CMakeLists.txt`
7. Existing tests related to the subsystem being changed

If `PROJECT_INDEX.md` is missing or outdated, create or update it before large work.

## Repository Map

Expected top-level layout:

* `app/`: executable entry point for the editor application
* `core/`: platform, logging, input, job system, event bus, console, CVar, VFS, profiling, service utilities
* `math/`: vectors, matrices, quaternions, transforms, bounds, rays, and geometric primitives
* `physics/`: rigid body simulation, collision, CCD, manifolds, queries, constraints, and joints
* `renderer/`: mesh loading, texture loading, shader system, command buffers, render settings, GL backend, shadows, post-processing, and asset systems
* `audio/`: runtime audio service and miniaudio integration
* `scripting/`: Lua runtime, Lua bindings, generated binding pipeline, coroutine helpers, timers, input bindings, scene bindings, and debug/DAP support
* `runtime/`: engine bootstrap, run loop, ECS/world, scene serialization, prefab serialization, scripting bridge, physics bridge, render preparation, service registry, and timer manager
* `editor/`: editor integration, editor camera, debug camera, command history, gizmos, and editor-facing workflows
* `assets/`: scripts, shaders, sample scenes, and sample runtime assets
* `tools/`: offline tools such as asset conversion, source audits, analysis helpers, and binding generation
* `tests/`: unit, integration, smoke, and benchmark tests
* `.github/workflows/`: CI configuration

## Architecture Rules

Maintain strict module boundaries.

Preferred dependency direction:

```text
app
  -> editor
  -> runtime
  -> renderer / physics / scripting / audio
  -> core / math
```

Rules:

* Do not introduce circular dependencies.
* Do not add sideways dependencies unless there is a clear architectural reason.
* Keep public headers self-contained.
* Do not expose SDL, OpenGL, Lua, ImGui, or ImGuizmo types through public engine headers unless explicitly justified.
* Keep platform-specific code behind platform or implementation boundaries.
* Keep OpenGL-specific code inside renderer implementation boundaries.
* Keep Lua-specific code inside scripting implementation boundaries.
* Keep editor-specific code out of runtime unless it is behind an explicit bridge.
* Prefer small focused changes over broad rewrites.
* Do not rewrite unrelated systems during a focused task.

## C++ Rules

Use C++20 only.

Required style and safety constraints:

* No exceptions.
* No RTTI.
* No `dynamic_cast`.
* No `typeid`.
* Prefer `noexcept` for engine-facing APIs where practical.
* Prefer explicit return values, status objects, or logged failure paths.
* Avoid hidden control flow.
* Avoid heap allocation in hot paths.
* Avoid global mutable state unless it is part of an explicit service or registry.
* Keep ownership clear.
* Prefer RAII for resources.
* Prefer fixed-size or preallocated storage in performance-sensitive systems.
* Avoid unnecessary virtual dispatch in hot runtime paths.
* Keep public APIs minimal and stable.

## LLVM, Clang, and clangd Indexing

The project should be configured so clangd can index it correctly.

The source of truth for clangd indexing is:

```text
build/compile_commands.json
```

This file must be generated by CMake with `CMAKE_EXPORT_COMPILE_COMMANDS=ON`.

Do not manually create or edit `compile_commands.json`.

Do not commit the `build/` directory.

Recommended local build directory:

```text
build/
```

Recommended project-level clangd config:

```yaml
CompileFlags:
  CompilationDatabase: build

Index:
  Background: Build

Diagnostics:
  ClangTidy:
    Add: []
```

Place this in:

```text
.clangd
```

### Preferred Windows clang-cl Configure Command

Use this when building on Windows with Visual Studio Build Tools or Visual Studio installed:

```powershell
cmake -S . -B build `
  -G Ninja `
  -DCMAKE_C_COMPILER=clang-cl `
  -DCMAKE_CXX_COMPILER=clang-cl `
  -DCMAKE_EXPORT_COMPILE_COMMANDS=ON `
  -DCMAKE_BUILD_TYPE=Debug
```

Then build:

```powershell
cmake --build build --parallel
```

Then verify:

```powershell
Test-Path .\build\compile_commands.json
```

Expected result:

```text
True
```

### Alternative Windows LLVM-MinGW Configure Command

Use this only when intentionally using LLVM-MinGW instead of the MSVC ABI toolchain:

```powershell
cmake -S . -B build `
  -G Ninja `
  -DCMAKE_C_COMPILER=x86_64-w64-mingw32-clang `
  -DCMAKE_CXX_COMPILER=x86_64-w64-mingw32-clang++ `
  -DCMAKE_EXPORT_COMPILE_COMMANDS=ON `
  -DCMAKE_BUILD_TYPE=Debug
```

Then build:

```powershell
cmake --build build --parallel
```

### Linux or macOS Clang Configure Command


## Runtime and ECS Rules

Runtime and ECS changes require extra care.

Before editing ECS/world code:

1. Read the public world headers.
2. Read current world implementation.
3. Read world, ECS, serialization, determinism, and stress tests.
4. Identify existing invariants before changing data layout or lifetime rules.

Rules:

* Preserve deterministic behavior.
* Preserve entity capacity assumptions unless explicitly changing them.
* Preserve persistent ID behavior.
* Do not break transform double-buffering.
* Do not silently change serialization format without migration or tests.
* Do not add per-frame heap allocations in hot ECS paths.
* Do not add editor-only assumptions into runtime data.

## Renderer Rules

Renderer changes require clear separation between frontend data and backend implementation.

Rules:

* Keep OpenGL details inside renderer implementation files.
* Avoid leaking raw GL handles into public APIs unless explicitly required.
* Keep command buffer construction separate from GL execution.
* Keep asset loading separate from render submission.
* Keep render settings centralized.
* Preserve forward fallback behavior when changing deferred rendering.
* Preserve transparency behavior when changing deferred rendering.
* Add tests for CPU-verifiable render preparation, command buffer, asset, shader, texture, and light-culling behavior.
* Treat GPU and headless execution carefully.

Before changing renderer code, inspect:

* `renderer/CMakeLists.txt`
* Renderer public headers
* Relevant renderer unit tests
* Relevant integration tests
* Shaders under `assets/shaders/`

## Physics Rules

Physics changes must preserve stability and determinism.

Rules:

* Keep simulation stepping deterministic where practical.
* Do not mix runtime ECS ownership with physics internals without an explicit bridge.
* Keep collision detection, contact generation, solver behavior, and queries testable.
* Add tests for edge cases, not only normal cases.
* Be careful with floating-point thresholds.
* Prefer explicit constants over unexplained magic numbers.
* Preserve existing behavior unless the change is intentional and tested.

Before changing physics code, inspect:

* `physics/CMakeLists.txt`
* Physics public headers
* Physics unit tests
* Runtime physics bridge code
* Determinism tests

## Scripting Rules

Lua scripting is an engine-facing API surface.

Rules:

* Do not break existing Lua API behavior without tests and documentation updates.
* Keep generated bindings and hand-written bindings clearly separated.
* Keep Lua-specific types out of unrelated public engine headers.
* Validate stack usage carefully.
* Preserve traceback and error reporting behavior.
* Preserve sandbox behavior unless explicitly changing security rules.
* Add or update tests for binding behavior, hot reload, coroutine behavior, timers, scene access, and runtime bridge behavior.

Before changing scripting code, inspect:

* `scripting/CMakeLists.txt`
* `scripting/src/scripting.cpp`
* Binding source files
* Binding generator tools
* `bindable_api.h`
* Existing scripting integration tests
* Lua scripts under `assets/`

## Editor Rules

Editor changes should not contaminate runtime architecture.

Rules:

* Keep editor-only behavior in `editor/`.
* Use explicit bridges for editor-runtime integration.
* Keep editor command history undoable and testable.
* Preserve play, pause, and stop semantics.
* Do not add editor-only assumptions to runtime systems.
* Add tests for command history or editor-facing non-UI logic when possible.

## Asset and Tooling Rules

Rules:

* Keep runtime asset loading and offline asset conversion separate.
* Keep generated files predictable.
* Prefer deterministic output for asset cooking and metadata.
* Do not add large binary assets unless explicitly requested.
* Do not add new third-party dependencies without confirmation.
* If adding tool behavior, add tests for parsing, dependency graph behavior, deterministic output, or error handling.

## Dependency Policy

Before adding a dependency:

1. Check whether existing code already provides the needed functionality.
2. Check whether the dependency is runtime, editor-only, test-only, or tool-only.
3. Prefer small, isolated dependencies.
4. Do not add dependencies to public headers unless necessary.
5. Do not add dependencies that require exceptions or RTTI in engine code.
6. Ask for confirmation before adding large third-party libraries.

## Performance Policy

For hot paths:

* Avoid heap allocation.
* Avoid string allocation.
* Avoid logging every frame.
* Avoid unordered containers unless justified.
* Avoid virtual dispatch unless justified.
* Avoid locking unless the scope is small and intentional.
* Prefer cache-friendly layouts.
* Prefer preallocation.
* Measure or test performance-sensitive changes when possible.

Hot-path areas include:

* ECS iteration
* Transform propagation
* Physics stepping
* Collision detection
* Render preparation
* Command buffer construction
* Asset streaming
* Input polling
* Job system scheduling

## Error Handling Policy

Use explicit failure paths.

Preferred options:

* `bool` with logging when failure detail is not needed
* Small status/result objects when failure detail matters
* Optional-like return values for missing data
* Assertions only for programmer errors and internal invariants

Avoid:

* Exceptions
* Silent failure
* Process termination for recoverable runtime errors
* Logging without returning an actionable failure state

## Known Risk Checks

When indexing or reviewing the repository, check for:

* Missing `build/compile_commands.json`
* Stale `.clangd` config
* Duplicate source files in CMake target lists
* Public headers leaking third-party implementation types
* Circular dependencies
* Runtime systems depending on editor code
* Per-frame heap allocation
* Missing tests for changed behavior
* Serialization changes without tests
* Lua API changes without tests
* Renderer code that cannot run in headless CI
* Platform-specific assumptions
* Generated files that are not deterministic
* Stale documentation

## Do Not Do

Do not:

* Delete source files unless explicitly asked.
* Rewrite the entire architecture in one pass.
* Add large third-party dependencies without confirmation.
* Add exceptions or RTTI.
* Hide build failures.
* Ignore failing tests.
* Modify unrelated systems.
* Commit changes unless explicitly asked.
* Convert public APIs to expose SDL, OpenGL, Lua, ImGui, or ImGuizmo types without justification.
* Treat generated code as the source of truth when generator inputs exist.
* Make editor-only behavior required for runtime execution.
* Commit `build/` or generated local build artifacts.
